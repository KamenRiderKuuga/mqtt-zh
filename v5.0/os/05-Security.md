# 5 安全性（非规范性）

## 5.1 概述

`MQTT` 是一个用于消息传输的传输协议规范，允许实现者选择网络、隐私、认证和授权技术。由于所选的确切安全技术将取决于具体场景，因此实现者有责任将适当的功能作为其设计的一部分。

`MQTT` 实现可能需要跟上不断变化的安全形势。

本章提供通用的实现指导，以便不限制可用的选择，因此是非规范性的。这不应减损其重要性。

强烈建议提供 `TLS` [RFC5246] 的 `Server` 实现使用 TCP 端口 8883（`IANA` 服务名称：secure-mqtt）。

解决方案提供商应该考虑许多威胁。例如：

- 设备可能被入侵
- `Client` 和 `Server` 中的静态数据可能被访问
- 协议行为可能产生副作用（例如"时序攻击"）
- 拒绝服务（DoS）攻击
- 通信可能被拦截、篡改、重路由或泄露
- 注入伪造的 `MQTT Control Packets` 

`MQTT` 解决方案通常部署在恶劣的通信环境中。在这种情况下，实现通常需要提供以下机制：

- 用户和设备的认证
- 访问 `Server` 资源的授权
- `MQTT Control Packets` 及其包含的应用数据的完整性
- `MQTT Control Packets` 及其包含的应用数据的隐私

除了技术安全问题外，还可能存在地理（例如美欧隐私盾框架 [USEUPRIVSH]）、行业特定（例如 PCI DSS [PCIDSS]）和监管方面的考虑（例如萨班斯-奥克斯利法案 [SARBANES]）。

## 5.2 `MQTT` 解决方案：安全性和认证

实现可能希望符合特定的行业安全标准，如 NIST 网络安全框架 [NISTCSF]、PCI-DSS [PCIDSS]、FIPS-140-2 [FIPS1402] 和 NSA Suite B [NSAB]。

关于在 NIST 网络安全框架 [NISTCSF] 中使用 `MQTT` 的指导可以在 `MQTT` 补充出版物《MQTT 和 NIST 改善关键基础设施网络安全框架》[MQTTNIST] 中找到。使用经过行业验证、独立验证和认证的技术将有助于满足合规性要求。

## 5.3 轻量级加密和受限设备

高级加密标准 [AES] 是最广泛采用的加密算法。许多处理器都有 AES 的硬件支持，但嵌入式处理器通常没有。加密算法 ChaCha20 [CHACHA20] 在软件中加密和解密速度快得多，但不如 AES 普遍可用。

ISO 29192 [ISO29192] 针对专门针对受限"低端"设备性能优化的加密原语提出了建议。

## 5.4 实现说明

在实现或使用 `MQTT` 时，需要考虑许多安全问题。以下内容不应被视为"检查清单"。

实现可能希望实现以下部分或全部目标：

### 5.4.1 `Server` 对 `Client` 的认证

`CONNECT` 报文包含用户名和密码字段。实现可以选择如何使用这些字段的内容。它们可以提供自己的认证机制，使用外部认证系统如 `LDAP` [RFC4511] 或 `OAuth` [RFC6749] 令牌，或利用操作系统认证机制。

`MQTT` v5.0 提供了如 4.12 节所述的增强认证机制。使用此功能需要 `Client` 和 `Server` 双方的支持。

以明文形式传递认证数据、混淆此类数据元素或不要求认证数据的实现应该意识到这可能导致中间人和重放攻击。5.4.5 节介绍了确保数据隐私的方法。

`Client` 和 `Server` 之间的虚拟专用网络（`VPN`）可以确保数据仅从授权的 `Client` 接收。

在使用 `TLS` [RFC5246] 的情况下，`Server` 可以使用 `Client` 发送的 `TLS` 证书来认证 `Client`。

实现可能允许通过从 `Client` 发送到 `Server` 的应用消息中的凭据进行认证。

### 5.4.2 `Server` 对 `Client` 的授权

如果 `Client` 已成功通过认证，`Server` 实现应该在接受其连接之前检查其是否被授权。

授权可以基于 `Client` 提供的信息，如用户名、`Client` 的主机名/IP 地址，或认证机制的结果。

特别是，实现应该检查 `Client` 是否被授权使用 `Client` 标识符，因为这可以访问 `MQTT` 会话状态（如 4.1 节所述）。此授权检查是为了防止一个 `Client`（意外或恶意地）提供已被其他 `Client` 使用的 `Client` 标识符的情况。

实现应该提供在 `CONNECT` 之后生效的访问控制，以限制 `Client` 发布到特定主题或使用特定主题过滤器订阅的能力。实现应该考虑限制对具有广泛范围的主题过滤器的访问，例如 # 主题过滤器。

### 5.4.3 `Client` 对 `Server` 的认证

`MQTT` 协议不是信任对称的。当使用基本认证时，`Client` 没有机制来认证 `Server`。某些形式的扩展认证确实允许相互认证。

在使用 `TLS` [RFC5246] 的情况下，`Client` 可以使用 `Server` 发送的 `TLS` 证书来认证 `Server`。从单个 IP 地址为多个主机名提供 `MQTT` 服务的实现应该注意 [RFC6066] 第 3 节中定义的 `TLS` 服务器名称指示扩展。这允许 `Client` 告诉 `Server` 它尝试连接的 `Server` 主机名。

实现可能允许通过从 `Server` 发送到 `Client` 的应用消息中的凭据进行认证。`MQTT` v5.0 提供了如 4.12 节所述的增强认证机制，可用于向 `Client` 认证 `Server`。使用此功能需要 `Client` 和 `Server` 双方的支持。

`Client` 和 `Server` 之间的 `VPN` 可以确保 `Client` 连接到预期的 `Server`。

### 5.4.4 应用消息和 `MQTT Control Packets` 的完整性

应用可以独立在其应用消息中包含哈希值。这可以在网络传输和静态存储时提供发布报文内容的完整性。

`TLS` [RFC5246] 提供哈希算法来验证通过网络发送的数据的完整性。

使用 `VPN` 连接 `Client` 和 `Server` 可以在 `VPN` 覆盖的网络部分提供数据完整性。

### 5.4.5 应用消息和 `MQTT Control Packets` 的隐私

`TLS` [RFC5246] 可以提供通过网络发送的数据加密。存在有效的 `TLS` 密码套件，其中包含不加密数据的 NULL 加密算法。为确保隐私，`Client` 和 `Server` 应该避免使用这些密码套件。

应用可以独立加密其应用消息的内容。这可以在网络传输和静态存储时提供应用消息的隐私。但这不会为应用消息的其他属性（如主题名称）提供隐私。

`Client` 和 `Server` 实现可以为静态存储的数据（如作为会话一部分存储的应用消息）提供加密存储。

使用 `VPN` 连接 `Client` 和 `Server` 可以在 `VPN` 覆盖的网络部分提供数据隐私。

### 5.4.6 消息传输的不可否认性

应用设计者可能需要考虑适当的策略来实现端到端的不可否认性。

### 5.4.7 检测 `Client` 和 `Server` 的入侵

使用 `TLS` [RFC5246] 的 `Client` 和 `Server` 实现应该提供能力，以确保在启动 `TLS` 连接时提供的任何 `TLS` 证书与连接的 `Client` 或被连接的 `Server` 的主机名相关联。

使用 `TLS` 的 `Client` 和 `Server` 实现可以选择提供检查证书吊销列表（CRL [RFC5280]）和在线证书状态协议（OCSP [RFC6960]）的能力，以防止使用已吊销的证书。

物理部署可能会将防篡改硬件与应用消息中特定数据的传输相结合。例如，电表可能具有嵌入式 GPS 以确保其不会在未经授权的位置使用。[IEEE8021AR] 是使用加密绑定标识符实现设备身份认证机制的标准。

### 5.4.8 检测异常行为

`Server` 实现可能会监控 `Client` 行为以检测潜在的安全事件。例如：

- 重复的连接尝试
- 重复的认证尝试
- 异常的连接终止
- 主题扫描（尝试发送或订阅许多主题）
- 发送无法投递的消息（主题没有订阅者）
- 连接但不发送数据的 `Client`

`Server` 实现可能会关闭违反其安全规则的 `Client` 的网络连接。

检测到不受欢迎行为的 `Server` 实现可能会根据 IP 地址或 `Client` 标识符等标识符实施动态阻止列表。

部署可能会使用网络级控制（如果可用）根据 IP 地址或其他信息实施速率限制或阻止。

### 5.4.9 处理不允许的 `Unicode` 码点

1.5.4 节描述了不应该包含在 `UTF-8` 编码字符串中的不允许 `Unicode` 码点。`Client` 或 `Server` 实现可以选择是否验证 `UTF-8` 编码字符串（如主题名称或属性）中没有使用这些码点。

如果 `Server` 不验证 `UTF-8` 编码字符串中的码点，但订阅 `Client` 进行验证，则另一个 `Client` 可能能够通过在主题名称或属性中使用不允许的 `Unicode` 码点发布消息，导致订阅 `Client` 关闭网络连接。本节建议可以采取的一些步骤来防止此问题。

当 `Client` 验证有效载荷与有效载荷格式指示符匹配而 `Server` 不验证时，可能会出现类似问题。这方面的考虑和补救措施与处理不允许的 `Unicode` 码点类似。

#### 5.4.9.1 使用不允许 `Unicode` 码点的考虑

实现通常会选择验证 `UTF-8` 编码字符串，检查是否没有使用不允许的 `Unicode` 码点。这可以避免实现困难，例如使用对这些码点敏感的库，也可以保护应用免于处理它们。

验证没有使用这些码点可以消除一些安全风险。可能存在使用日志文件中的控制字符来掩盖日志条目或混淆处理日志文件的工具的安全攻击。`Unicode` 非字符通常用作特殊标记，允许它们进入 `UTF-8` 编码字符串可能允许此类攻击。

#### 5.4.9.2 发布者和订阅者之间的交互

应用消息的发布者通常期望 `Server` 将消息转发给订阅者，并且这些订阅者能够处理消息。

以下是发布 `Client` 可能导致订阅 `Client` 关闭网络连接的一些情况。考虑以下情况：

- `Client` 使用包含不允许 `Unicode` 码点之一的主题名称发布应用消息。
- 发布 `Client` 库允许在主题名称中使用不允许的 `Unicode` 码点，而不是拒绝它。
- 发布 `Client` 被授权发送该发布。
- 订阅 `Client` 被授权使用与主题名称匹配的主题过滤器。注意，不允许的 `Unicode` 码点可能出现在主题名称中与主题过滤器中的通配符匹配的部分。
- `Server` 将消息转发给匹配的订阅者，而不是断开发布者的连接。

在这种情况下，订阅 `Client` 可能：

- 关闭网络连接，因为它不允许使用不允许的 `Unicode` 码点，可能在这样做之前发送 `DISCONNECT`。对于 `QoS` 1 和 `QoS` 2 消息，这可能导致 `Server` 再次发送消息，导致 `Client` 再次关闭网络连接。
- 通过在 `PUBACK`（`QoS` 1）或 `PUBREC`（`QoS` 2）中发送大于或等于 0x80 的原因码来拒绝应用消息。
- 接受应用消息但因为其包含不允许的 `Unicode` 码点之一而无法处理它。
- 成功处理应用消息。

`Client` 关闭网络连接的可能性可能在发布者使用不允许的 `Unicode` 码点之一之前不会被发现。

#### 5.4.9.3 补救措施

如果主题名称或其他属性中可能包含不允许的 `Unicode` 码点并传递给 `Client`，解决方案所有者可以采用以下建议之一：

1) 将 `Server` 实现更改为通过发送大于或等于 0x80 的原因码或关闭网络连接来拒绝包含不允许 `Unicode` 码点的 `UTF-8` 编码字符串。

2) 将订阅者使用的 `Client` 库更改为能够容忍使用不允许码点的库。只要继续执行协议，`Client` 可以处理或丢弃包含不允许 `Unicode` 码点的 `UTF-8` 编码字符串的消息。

### 5.4.10 其他安全考虑

如果 `Client` 或 `Server` `TLS` 证书丢失或被认为可能已受损，应该将其吊销（使用 CRL [RFC5280] 和/或 OCSP [RFC6960]）。

丢失或被认为已受损的 `Client` 或 `Server` 认证凭据（如用户名和密码）应该被吊销和/或重新颁发。

在长时间连接的情况下：

- 使用 `TLS` [RFC5246] 的 `Client` 和 `Server` 实现应该允许会话重新协商，以建立新的加密参数（替换会话密钥、更改密码套件、更改认证凭据）。
- `Server` 可以关闭 `Client` 的网络连接并要求它们使用新凭据重新认证。
- `Server` 可以使用 4.12.1 节中描述的机制定期要求其 `Client` 重新认证。

受限设备和受限网络上的 `Client` 可以使用 `TLS` [RFC5246] 会话恢复，以减少重新连接 `TLS` [RFC5246] 会话的成本。

连接到 `Server` 的 `Client` 与连接到同一 `Server` 并有权在相同主题上发布数据的其他 `Client` 具有传递信任关系。

### 5.4.11 `SOCKS` 的使用

`Client` 实现应该注意到，某些环境需要使用 `SOCKSv5` [RFC1928] 代理来进行出站网络连接。某些 `MQTT` 实现可以通过使用 `SOCKS` 利用替代的安全隧道（例如 `SSH`）。如果实现选择使用 `SOCKS`，它们应该支持匿名和用户名、密码认证的 `SOCKS` 代理。在后一种情况下，实现应该注意 `SOCKS` 认证可能以明文形式发生，因此应该避免使用相同的凭据连接到 `MQTT Client`。

### 5.4.12 安全配置文件

实现者和解决方案设计者可能希望将安全性视为可应用于 `MQTT` 协议的一组配置文件。下面给出了分层安全层次结构的示例。

#### 5.4.12.1 明文通信配置文件

使用明文通信配置文件时，`MQTT` 协议在开放网络上运行，没有额外的安全通信机制。

#### 5.4.12.2 安全网络通信配置文件

使用安全网络通信配置文件时，`MQTT` 协议在具有安全控制（例如 `VPN` 或物理安全网络）的物理或虚拟网络上运行。

#### 5.4.12.3 安全传输配置文件

使用安全传输配置文件时，`MQTT` 协议在物理或虚拟网络上运行，并使用 `TLS` [RFC5246] 提供认证、完整性和隐私。

`TLS` [RFC5246] `Client` 认证可以用于补充或替代由用户名和密码字段提供的 `MQTT Client` 认证。

#### 5.4.12.4 行业特定安全配置文件

预计 `MQTT` 协议将被设计到行业特定的应用配置文件中，每个配置文件定义威胁模型和用于解决这些威胁的特定安全机制。特定安全机制的建议通常取自现有工作，包括：

[NISTCSF] NIST 网络安全框架

[NIST7628] NISTIR 7628 智能电网网络安全指南

[FIPS1402] 加密模块安全要求（FIPS PUB 140-2）

[PCIDSS] PCI-DSS 支付卡行业数据安全标准

[NSAB] NSA Suite B 密码学
